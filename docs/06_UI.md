## 1. Framework Decision & Rationale

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Meta-framework | **Next.js 14+ (App Router)** | RSC streaming for analytics dashboards, file-system routing matches our page taxonomy, server actions for form mutations |
| Language | **TypeScript 5.x (strict mode)** | End-to-end type safety with backend Pydantic models via OpenAPI codegen |
| Styling | **Tailwind CSS v3** | Enforces design-token consistency via `tailwind.config.ts`, eliminates dead CSS in bundle, utility-first fits component-library pattern |
| Primitives | **shadcn/ui** | Unstyled, accessible, Radix-based components we own as source; no lock-in, full customization via CSS vars |
| Server state | **TanStack Query v5** | Caching, deduplication, optimistic updates for API-heavy content pages; SSR hydration via `initialData` |
| Client state | **Zustand** | Minimal boilerplate for interactive panels (draft editor, notification panel, theme toggle); no provider nesting |
| Form handling | **React Hook Form + Zod** | Aligns with LLD validation schemas; Zod schemas generated from Pydantic models |
| Charts | **Chart.js via react-chartjs-2** (dynamic import, `ssr: false`) | Audience is technical; Chart.js is well-known and avoids WebGL overhead of Recharts/ECharts |
| Auth | **NextAuth.js v5 (Auth.js)** | Built-in OAuth providers (GitHub, Google), JWT session strategy for BFF pattern, middleware for route protection |

### 1.1 Server Components First

```
RSC default boundary
├── app/page.tsx (dashboard — fetches aggregated data, streams panels)
├── app/content/page.tsx (content list — fetches paginated items)
├── app/kb/page.tsx (knowledge base — fetches items + tags)
├── app/analytics/page.tsx (fetches metrics, chart data as props)

'use client' boundary (interactive islands)
├── app/drafts/[id]/page.tsx (draft editor — real-time preview, revision history)
├── app/content/components/ContentForm.tsx (create/edit forms)
├── app/settings/page.tsx (profile editor, preferences)
├── components/dashboard/AnalyticsChart.tsx (Chart.js wrapper)
├── components/notifications/NotificationPanel.tsx (polling + real-time)
```

Every page that displays data is a Server Component. Client interactivity is pushed to the deepest possible leaf. No page-level `'use client'`.

### 3.2 Coding Convention

- **No arbitrary values**: Never use `h-[200px]` or `text-[#abc]`. Extend the config.
- **Composition > custom CSS**: Use utility classes + `cn()` helper for conditional styles. Only write custom CSS for keyframe animations and complex pseudo-class interactions.
- **Surface colors**: Use `bg-surface`, `bg-surface-elevated`, `bg-surface-hover` instead of literal gray tones — these switch with theme automatically.

```tsx
// ✅ Correct
<div className="bg-surface-elevated border border-border rounded-lg p-6 shadow-sm">

// ❌ Avoid
<div className="bg-white dark:bg-gray-800 border border-gray-200 dark:border-gray-700 rounded-lg p-6 shadow-sm">
```

## 4. Theme System

### 4.1 Architecture

```
User preference ──► localStorage ──► <html class="dark|light">
                        ▲
System preference ──►  matchMedia('prefers-color-scheme: dark')
```

1. On first visit, check `localStorage` for saved theme
2. If no saved preference, respect `prefers-color-scheme: dark` media query
3. Save toggle choice to `localStorage`
4. Apply `.dark` class to `<html>` element
5. Flash prevention: inline critical CSS in `<head>` via Next.js `script` with `dangerouslySetInnerHTML` that runs before paint

### 4.2 Flash Prevention Script

```tsx
// app/providers.tsx
const themeScript = `
  (function() {
    const stored = localStorage.getItem('theme');
    const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
    if (stored === 'dark' || (!stored && prefersDark)) {
      document.documentElement.classList.add('dark');
    }
  })();
`;
```

This script executes synchronously in the `<head>` before any paint, preventing the flash of incorrect theme.

### 4.3 ThemeProvider (Client Component)

```tsx
// components/theme/ThemeProvider.tsx
'use client';

import { createContext, useContext, useEffect, useState } from 'react';

type Theme = 'light' | 'dark';

const ThemeContext = createContext<{
  theme: Theme;
  toggle: () => void;
}>({ theme: 'light', toggle: () => {} });

export function ThemeProvider({ children }: { children: React.ReactNode }) {
  const [theme, setTheme] = useState<Theme>('light');
  const [mounted, setMounted] = useState(false);

  useEffect(() => {
    setMounted(true);
    const stored = localStorage.getItem('theme') as Theme | null;
    const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
    const initial = stored || (prefersDark ? 'dark' : 'light');
    setTheme(initial);
    document.documentElement.classList.toggle('dark', initial === 'dark');
  }, []);

  const toggle = () => {
    const next = theme === 'light' ? 'dark' : 'light';
    setTheme(next);
    localStorage.setItem('theme', next);
    document.documentElement.classList.toggle('dark', next === 'dark');
  };

  if (!mounted) return <>{children}</>;

  return (
    <ThemeContext.Provider value={{ theme, toggle }}>
      {children}
    </ThemeContext.Provider>
  );
}

export const useTheme = () => useContext(ThemeContext);
```

## 7. Dark Theme

### 7.1 Implementation

Dark mode uses the `.dark` class on `<html>`, driving CSS custom properties to new values. The Tailwind `darkMode: 'class'` strategy means `dark:bg-neutral-900` works, but we avoid explicit dark prefixes — the CSS variables handle theme switching automatically.

```tsx
// ✅ Correct — CSS variable handles both themes
<div className="bg-surface-elevated border border-border">

// ❌ Avoid — defeats the purpose of design tokens
<div className="bg-white dark:bg-gray-800 border border-gray-200 dark:border-gray-700">
```

### 7.2 Component Dark Mode

When a component needs different visual treatment per theme (e.g., a card that is darker in dark mode), the token approach handles it:

- `bg-surface-elevated`: white in light mode, `#25262b` in dark mode
- `border-border`: `#dee2e6` in light mode, `#373a40` in dark mode
- `text-secondary`: `#495057` in light mode, `#909296` in dark mode

### 7.3 Media Handling

Charts and code blocks need explicit dark-aware handling:

- Charts: pass `data.canvas` with theme-aware grid colors via `chartjs-plugin-dark`
- Markdown rendered content: prose styles use CSS variables via Tailwind typography plugin `prose-invert`

## 15. Error States

### 15.1 Error Boundary

```tsx
// components/ErrorBoundary.tsx
'use client';

interface ErrorBoundaryProps {
  fallback?: React.ReactNode;
  children: React.ReactNode;
}

interface ErrorBoundaryState {
  hasError: boolean;
  error?: Error;
}

export class ErrorBoundary extends React.Component<ErrorBoundaryProps, ErrorBoundaryState> {
  constructor(props: ErrorBoundaryProps) {
    super(props);
    this.state = { hasError: false };
  }

  static getDerivedStateFromError(error: Error) {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, info: React.ErrorInfo) {
    console.error('ErrorBoundary caught:', error, info);
    // Optionally send to logging service
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback || <DefaultErrorFallback error={this.state.error} />;
    }
    return this.props.children;
  }
}
```

## 16. Accessibility

### 16.1 Standards Target

**WCAG 2.2 Level AA** — measured via axe-core in CI and manual keyboard audit.

### 16.2 Requirements Checklist

| Requirement | Implementation |
|-------------|---------------|
| Skip navigation | `<SkipToContent />` as first focusable element |
| Landmark regions | `<header>`, `<nav>`, `<main>`, `<aside>` for sidebar/drawers |
| Heading hierarchy | h1 per page → h2 for sections → h3 for card titles. No skipping levels |
| Focus indicators | `focus-visible:ring-2 focus-visible:ring-primary` on all interactive elements |
| Color contrast | All text/background combos meet 4.5:1 (AA) or 3:1 (large text); tested via design tokens, not ad-hoc colors |
| Keyboard navigation | All interactive elements reachable and operable via Tab/Shift+Tab/Enter/Space/Arrow keys |
| ARIA live regions | `aria-live="polite"` for dynamic content updates; `aria-live="assertive"` for errors |
| Form labels | Every `<Input>`, `<Select>`, `<Textarea>` has associated `<Label>` |
| Status announcements | `role="status"` on toast notifications |
| Reduced motion | `prefers-reduced-motion: no-animation` disables skeleton pulse and fade-in animations |
| Screen reader announcements | `aria-describedby` for field errors; `aria-current="page"` on active nav item |
| Touch targets | Minimum 44×44px on mobile for all interactive elements |

## 19. Performance Budget

### 19.1 Targets

| Metric | Target | Measurement |
|--------|--------|------------|
| LCP | ≤ 2000ms (mobile 4G p75) | Next.js Web Vitals |
| FID | ≤ 100ms | Next.js Web Vitals |
| CLS | ≤ 0.1 | Next.js Web Vitals |
| JS per route | ≤ 150 KB gzip | `next/bundle-analyzer` |
| Initial HTML | ≤ 50 KB | Response size |
| API response | ≤ 500ms p95 | Server logs |

### 19.2 Optimization Techniques

| Technique | Application |
|-----------|------------|
| **RSC streaming** | Dashboard panels, analytics sections — show shell immediately |
| **Dynamic imports** | Charts (`next/dynamic` with `ssr: false`), markdown renderer, CodeMirror |
| **Image optimization** | `next/image` for all illustrations; `next/font` for Google Fonts |
| **Bundle splitting** | Draft editor on separate route, lazy-load editor libraries |
| **Route prefetch** | `<Link>` prefetches visible links on viewport entry |
| **Data caching** | TanStack Query `staleTime` prevents re-fetch for dashboard on tab switch |
| **Code splitting** | shadcn components tree-shake via named exports; barrel files only from `components/ui/` |

## 20. Testing Strategy

### 20.1 Test Levels

| Level | Tool | Scope | Frequency |
|-------|------|-------|-----------|
| Unit | Vitest + Testing Library | Components, hooks, utils | Every commit |
| Integration | Vitest + MSW | Pages with API mocking | Every PR |
| E2E | Playwright | Critical flows (auth, content CRUD) | CI pipeline |
| A11y | axe-core (via Playwright) | Every page | CI pipeline |
| Visual | Storybook + Chromatic | Component regressions | Every PR |

### 20.2 What to Test

- **Every shadcn usage pattern**: Verify it renders, responds to clicks, shows/hides as expected
- **Every empty state**: Component renders correctly when data prop is empty array/null
- **Every error boundary**: Wrap async data fetch, verify fallback renders
- **Every form**: Submit valid data, submit invalid data, verify error messages
- **Theme toggle**: Click toggle, verify `.dark` class on `<html>`, verify localStorage
- **Keyboard navigation**: Tab through every interactive element on each page
- **Loading states**: Verify skeleton renders while data is null, content replaces skeleton after fetch
